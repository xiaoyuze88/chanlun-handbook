# /chanlun-compare

对同一个缠论问题，分别给出"通用回答"和"知识库回答"，然后逐维度对比评测。用于：

1. **横向比较回答效果**：比较通用回答和知识库回答的效果
2. **反向验证**：发现知识库中可能的不准确之处

## 用法

```
/chanlun-compare <问题>
```

示例：

```
/chanlun-compare 缠论第一类买点的定义
/chanlun-compare 走势必完美的含义是什么
```

## 执行流程

### Step 1 — 通用回答（Baseline）

**不读项目文件、不查知识库**，仅凭训练记忆回答用户问题。

输出格式：

```
【通用回答】

<用自己的话回答，正常风格即可>
```

### Step 2 — 知识库回答

按项目 CODEBUDDY.md 中的"缠论知识库查询规则"完整执行：

1. **Step 0 问题分类与路由**：先判断问题类型（定义类/定理类/操作类/溯源类/关系类/综合类），再决定默认入口
2. **Step 1 附录低成本命中检测**：按题型锁定候选附录（A~E），做词条名/标题/规则ID级别的检索
3. **Step 2 graphify query 定位**：从问题提取2-4个核心中文关键词，运行 Python BFS 查询 `graphify-out/graph.json`，获取相关文件列表
4. **Step 3 阅读正文**：读取定位到的 organized-v2 文件，检查前置知识注释
5. **Step 4 组织回答**：先给结论再给推导，日常语言，可举例类比

输出格式：

```
【知识库回答】

<按规范组织的回答>

📎 引用来源
- <文件路径> § <段落标题>
```

### Step 3 — 结构化对比

对两个版本逐维度 **1-5 分打分**（1=极差，5=优秀），并做差异标注。

**评分标准**：

| 分数 | 含义 |
|------|------|
| 1 | 严重错误或完全缺失 |
| 2 | 有内容但存在明显错误/遗漏 |
| 3 | 基本正确但不够精确/完整 |
| 4 | 正确且较完整，小瑕疵 |
| 5 | 准确、完整、精确、有原文支撑 |

**评测维度与评分指引**：

| 维度 | 1分 | 3分 | 5分 |
|------|-----|-----|-----|
| 准确性 | 有事实性错误 | 大方向正确但细节有误 | 与原文完全一致 |
| 完整性 | 缺少核心要素 | 覆盖主要点但遗漏边界/例外 | 充要条件完整、边界清晰 |
| 精确性 | 模糊描述，无精确条件 | 有条件但不够充要 | 充要条件明确，可形式化 |
| 可操作性 | 无法直接执行 | 有步骤但不够具体 | 步骤+触发+判断树完整 |
| 原文溯源性 | 无出处 | 提及来源但未精确到课文 | 精确到课文编号+行号 |
| 边界/例外处理 | 未提及 | 提及但不系统 | 系统覆盖所有已知边界 |

**输出格式**：

```
## 对比评测

| 维度 | 通用回答 | 知识库回答 | 差值 |
|------|---------|-----------|------|
| 准确性 | <1-5> <一句话理由> | <1-5> <一句话理由> | <+N/-N> |
| 完整性 | | | |
| 精确性 | | | |
| 可操作性 | | | |
| 原文溯源性 | | | |
| 边界/例外处理 | | | |
| **总分** | **/30** | **/30** | **<差值>** |

### 差异标注
- [差异1] 通用版说X，知识库版说Y → 原因分析 → 哪方更准确
- [差异2] ...

### 知识库疑似问题
- <如果知识库回答中有与原文不符或可疑之处，标注出来>
- <如果通用回答中有知识库未覆盖的合理观点，也标注出来>
```

### Step 4 — 归档

将完整评测记录追加到 `~/Sites/private/chzhshch-108-plus/docs/eval/EVAL_LOG.md`。

如果该文件不存在，先创建并写入表头：

```markdown
# 缠论双路对比评测记录

> 通过 `/chanlun-compare` 命令生成。每次执行追加一条记录。
>
> 用途：量化知识库价值 + 反向验证知识库准确性

---
```

每次评测追加的格式：

```markdown
## #N <问题简述> | YYYY-MM-DD

**问题**：<完整问题>

**通用回答**：
<Step 1 输出>

**知识库回答**：
<Step 2 输出>

**对比评测**：
<Step 3 输出>

**引用来源**：

- <知识库查询涉及的文件路径及段落标题>

---
```

编号 #N 从 1 开始递增，取 EVAL_LOG.md 中已有记录的最大编号 +1。

## 注意事项

- Step 1（通用回答）**绝不能读项目文件**，否则失去 baseline 价值
- Step 2（知识库回答）**必须走完整流程**，不能跳步
- 对比维度中"精确性（充要条件）"是知识库最核心的优势维度，重点关注
- "知识库疑似问题"是反向验证的关键输出，不可遗漏
