# 缠论概念复核记录

## 目的

对 organized-v2 中的核心概念进行逐一复核，确认表述准确、无歧义，并对易混淆边界加注释。

## 状态说明

- ✅ 已复核，无问题
- 🔧 已复核，已修改（附修改说明）
- ⏳ 进行中
- ⬜ 待复核

## 主会话调度说明

复核分为两个优先级，agent 数量不同，流程相同：

- **P0（6 agent）**：较复杂或基础性强的概念，并行发起 6 个 subagent
- **P1（3 agent）**：相对独立的概念，并行发起 3 个 subagent（kimi-k2.5-ioa / glm-5.1-ioa / minimax-m2.7-ioa）

### 主会话职责

1. **准备任务包**：用 graphify query（`graphify-out/graph.json`）定位相关 organized-v2 文件，填入 subagent prompt
2. **并行发起 subagent**：在单条消息中同时发起，`run_in_background: true`
   - **P0 使用以下 6 个模型**：
     - `glm-5.1-ioa`
     - `minimax-m2.7-ioa`
     - `kimi-k2.5-ioa`
     - `gpt-5.4`
     - `gemini-3.1-pro`
     - `claude-sonnet-4.6-1m`
   - **P1 使用以下 3 个模型**：
     - `kimi-k2.5-ioa`
     - `glm-5.1-ioa`
     - `minimax-m2.7-ioa`
3. **等待全部结论**：通过 SendMessage 接收各 subagent 回传
4. **汇总评估**：横向对比结论，找共识与分歧；按以下规则处理：
   - **多 agent 共识问题**（≥2 份）：直接纳入修改清单
   - **单 agent 独家问题**：主会话必须独立复核（读原文验证），不可直接采信；复核结论记入修改记录（含提出方、主会话判断、原文依据）
   - **🔴 必须修复**的问题：无论共识度如何，主会话均独立读原文确认后再执行
5. **向用户汇报**：综合判断是否需要改、改哪里、怎么改，等用户确认后写入文件
6. **更新记录**：
   - 在 **REVIEW.md 复核列表**更新状态、结论摘要、日期
   - 将完整修改记录（含 agent 报告、共识度、主会话复核）写入 **[CHANGELOG.md](./CHANGELOG.md)**

### subagent 职责

执行下方"复核流程（每个概念）"的步骤1-4，只读不写，通过 SendMessage 汇报结论。

## 复核流程（每个概念）

> 本流程同时作为 subagent 任务描述，由主会话填入具体概念信息后派发。

### 步骤1：读 organized-v2 原文

读取以下文件：`{文件列表}`（主会话已通过 graphify query 定位，直接读取）

- 先检查文件头的 `<!-- 前置知识：... -->` 注释，若有则先读前置知识文件
- 识别该概念的定义、所有相邻概念、已有注释

### 步骤2：定位可疑点

找同一文件内两个或多个相邻概念，判断它们之间的关系是否说清楚了。重点看：
- 充要条件 vs 必要条件
- 触发条件 vs 内部结构
- 定义 vs 操作规则

### 步骤3：溯源原文（必做）

- **第一层**：查 organized-v2 文件末尾的"出处索引"表格，找到对应课文编号（如"课54"），直接读 `archived/108/` 对应课文，与 organized-v2 表述逐一对比
  - 文件命名规则：`{4位序号}-{hash}-{3位课号}.md`，如课54 = `0535-xxx-054.md`
  - 用 `ls /Users/xyz/Sites/private/chzhshch-108-plus/archived/108/ | grep "{序号}"` 定位文件
- **第二层（可选）**：若第一层对比后仍有不确定，查 `archived/graphify-out-108/graph.json` 辅助定位相关课文
- **第三层（可选）**：如需查中间整理过程，可查 `archived/organized-v1/` 对应内容

> 溯源路径：organized-v2 → archived/organized-v1 → archived/108（原始博客）

### 步骤4：汇报结论

完成后用 SendMessage 工具将结论发送给 `main`，格式如下：

```
Model: {模型ID}

## 总体评估
[整体准确性，主要关注点，做得好的地方]

## 问题清单

### 🔴 必须修复
- **位置**：文件名 § 章节名（或行号）
- **问题**：[具体描述，引用 organized-v2 原文 vs archived/108 原文对比]
- **建议**：[改成什么样，原文依据]

### 🟡 建议修复
（同上格式）

### 💭 小改进
（同上格式）

## 无问题确认
[若某部分核实无误，明确说明]

## 溯源记录
[列出读取的 archived/108 文件路径及关键原文段落]
```

**规则**：只读不写，不修改任何文件；每个问题必须有原文依据，不凭感觉判断。
